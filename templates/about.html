About HANSEL

HANSEL is a companion to GRETIL. Its goal is to continue the free and open exchange of academic-level Sanskrit e-texts, with a focus on diplomatic transcriptions of printed editions.

Name and logo

"Human-accessible" means that material will always exist in forms that the greatest number of students and scholars can use directly, such as plain-text and HTML.

"NLP-ready" means that the data will also be structured in anticipation of computational applications.

The haṃsa (swan, or bar-headed goose) symbolizes not only grace but also discernment (viveka). A world away, in the Grimms' tale, Hänsel and Gretel succeed against the odds by working together.

Principles

Keep submission as easy as sending an email, with material in whatever format.

Make sure metadata always clearly indicates sources and contributions.

Honor different goals by maintaining multiple versions of files, using a three-tier system to indicate progression toward an explicit data standard:

  • Tier I: Whichever initial version the contributor chooses
  • Tier II: Passing automatic validation for HANSEL's structural data model
  • Tier III: High-fidelity diplomatic correspondence to source edition

A solo creator working open-source is best to get things moving, and if the project works, control can be passed on to a rotating committee at a later date.

How It Works

HANSEL consists of two coordinated Git repositories: one for data and basic hosting with GitHub Pages, and one for the primary web app with extra features. Using Git takes care of fine-grained versioning. The public GitHub platform enables open-source access to everything, including PR-based change history.

The web app is written in Python and Javascript and runs on a Digital Ocean server which holds an up-to-date clone of the data repo.

Data is progressed through the system in the following manner:

  1. Users submit material by email. agree to share it under the CC BY-NC-SA 4.0 license, and communicate what remains to be done and how involved they'd like to be in processing.
  2. Metadata template is filled out.
  3. New text and metadata are uploaded and published on the site immediately.
  4. Work begins on a Tier II version.
  5. Tier II version and updated metadata are published.
  6. Time permitting, a Tier III version is also prepared and published.

Contribution

HANSEL's contribution system has three stages, with correspondingly different file formats:

  1. Pre-Draft — any file format (.doc, .pdf, etc.), for email correspondence and preliminary local processing
  2. Draft — plain-text (.txt, with custom markup), for provisional display while progressing toward publication
  3. Publication — XML (.xml, with minimal TEI markup), for web display and scholarly citation

Pre-Draft material is moved manually toward Draft format. Once the material passes a set of automated checks — run locally by technical team members and also automatically on GitHub — it is suited for Draft presentation. Such mature Draft material (.txt) can then be automatically transformed into Publication material (.xml), and vice versa. This allows for continued improvement of material without the need to work in XML.

Presentation

The website you are reading dynamically serves both Draft and Publication material alike, with Drafts marked as such. Publication material is rendered by default on the website with .html transforms, but users can also opt for a .txt transform or the underlying .xml. Cumulative downloads, both of the entire collection, or of filtered search sets, are also always at the ready. The server also takes care of metadata search and basic transliteration.
