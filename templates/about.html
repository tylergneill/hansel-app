About

HANSEL is a companion to GRETIL. Its goal is to continue the open exchange of academic-level Sanskrit e-texts, with a focus on diplomatic transcriptions of printed editions produced using OCR.

Name and logo

"Human-accessible" means that material will always exist in forms that the greatest number of students and scholars can use directly, such as plain-text and HTML.

"NLP-ready" means that the data will also be structured in anticipation of computational applications.

The haṃsa (swan, or bar-headed goose) symbolizes not only grace but also discernment (viveka). A world away, in the Grimms' tale, Hänsel and Gretel succeed against the odds by working together.

Additional Principles

Submitting material should be easy.

Metadata should show provenance.

All data and code should be versioned.

Originals and derivatives can coexist.

Multi-Tier Concept

In order to offer original data, with all its idiosyncracies, while also supporting development toward a particular standard, HANSEL uses a three-tier system to indicate progression toward its own data model:

  • Tier I: Whichever initial version the contributor chooses
  • Tier II: Passing automatic structural validation for HANSEL's data model
  • Tier III: High-fidelity diplomatic correspondence to source edition

Who's Who

A solo creator working open-source is best to get things moving, and if the project works, control can be passed on to a rotating committee at a later date.

Contribution How-To

  1. Tier I (1–5 days): Contributor writes in to `hanselrepository` at Gmail, usually with existing text material to share. CC BY-NC-SA 4.0 license is explained and remaining work is divvied up. Maintainer posts material ASAP.
  2. Tier II (2–4 weeks): Maintainer (or more rarely, contributor) manually adapts text to HANSEL data model. Once text passes validation and metadata has been improved, second-version material is posted.
  3. Tier III (3–6 months): Material is further reworked to improve diplomatic correspondence to source edition, then posted in consultation with active collaborators.

File Types

Tier I allows for any text-based file type, whereas Tiers II and III primarily use plaintext.

The HANSEL data model foresees automatic transformation from plaintext to TEI-XML. This XML is then transformed to HTML which is then the default for rendering on the website (browsers are happiest when handling HTML, for memory management, etc.) All of these options can be downloaded as cumulative files from the website or from GitHub.

Metadata is stored primarily in Markdown, which renders well on GitHub. For display on the website, there are also HTML transforms. On the backend, the web app uses a JSON transform. All formats are downloadable in bulk.

All non-primary formats are semi-automatically regenerated in bulk after changes to primary formats.

Under the Hood

HANSEL consists of two coordinated Git repositories: one for data, and one for the web app. Using Git takes care of fine-grained versioning. Public GitHub repos enable open-source access to everything, including PR-based change history.

The web app is written in Python and Javascript and runs on a Digital Ocean server which holds an up-to-date clone of the data repo. The data repo also contains Python management scripts run periodically and locally.
